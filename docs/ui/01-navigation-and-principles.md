# 全局导航与交互原则

> 日期：2026-03-27
> 用途：定义所有页面共享的外壳、导航、信息层级和跨页交互规则

## 1. 设计定位

SwarmMind 的 UI 应呈现为“项目控制面 + Team 协作面 + 治理面”的组合，而不是 IM 工具加一点项目管理。

全局体验应体现四个原则：

- `Project` 是唯一正式工作边界
- `TeamInterfaceAgent` 是项目入口，而不是普通聊天 Bot
- 审批是工作流中的显式暂停节点
- Artifact、Run、Audit 都必须能被看见和追溯
- 工作台是每个用户自己的驾驶舱，只按权限暴露不同范围的数据

```text
|------------------------------|

|------------------------------|
| [头像] Kr Li            v    |

```

### 2.2 导航原则

- `工作台` 是首页，不等于项目页。
- `+ 新建对话` 是立即动作，进入轻量 `ChatSession`。
- `项目` 是分组，不只是单个入口。
- `最近` 只承接个人最近访问，不与项目列表混排。
- 底部账户区固定，不随中部滚动。

## 3. 全局页面骨架

```text
+------------------------------------------------------------------------------------------------------------------+
| 侧边栏                                                                 | 顶栏：面包屑 / 搜索 / 视图切换 / 主动作      |
+--------------------------------+---------------------------------------------------------------------------------+
| 导航与项目分组                 | 页面主体                                                                         |
|                                |----------------------------------------------------------------------------------|
|                                | 左：列表 / 主内容 / 看板 / 时间线                                                |
|                                | 中：主工作区                                                                     |
|                                | 右：详情 / 上下文 / 审批 / 运行状态                                              |
|                                |                                                                                  |
|                                | 底：流式日志 / 状态栏 / 快捷输入                                                 |
+--------------------------------+---------------------------------------------------------------------------------+
```

说明：

- 不是每个页面都使用三栏，但页面结构尽量共享同一栅格逻辑。
- 右栏默认承担“详情、治理、上下文”角色。
- 全局主动作始终在顶栏右上角。

### 4.2 全局命令面板

建议提供 `Cmd/Ctrl + K` 命令面板，用于：

- 搜索项目
- 搜索最近会话
- 跳转到审批
- 新建项目
- 新建对话
- 切换到技能中心

### 4.3 全局状态条

右上角建议保留轻量系统状态：

- 连接状态
- 当前组织
- 待审批数量
- 正在运行的项目数

## 5. 跨页交互原则

### 5.1 ChatSession 到 Project

- 轻量聊天支持快速探索。
- 一旦用户要共享、审批、挂 Team、进入工作流，应引导 `提升为 Project`。
- 提升动作不是“保留整段聊天”，而是把语义摘要写入项目启动资料。

### 5.2 Project 内的统一入口

- 项目内聊天入口默认连接 `TeamInterfaceAgent`。
- 用户看到的是“项目协作台”，而不是一个孤立对话窗口。
- 入口消息会影响看板、时间线、风险和 artifact，而不是只追加聊天记录。

### 5.3 审批中断

- 审批一旦触发，项目界面必须显示暂停状态。
- 审批结果要能回流到原项目页并自动恢复工作流。

### 5.4 技能生效路径

- 能力配置在 `技能中心`。
- 能力生效反馈要回到 `项目` 与 `审批` 页面中可见。
- 用户不能在多个设置页之间来回猜测绑定是否成功。

### 5.5 工作台即驾驶舱

- 不单独拆“高层驾驶舱”页面。
- 所有用户都从 `工作台` 进入系统。
- 差异只来自权限和角色：
  - 普通成员看到与自己相关的项目、审批和运行摘要
  - 项目负责人看到自己负责项目的更完整健康度
  - 高权限管理者看到组织级聚合、跨项目风险和 Team 负载
- 因此工作台必须支持基于权限的卡片增减和数据范围扩展，而不是换一套新页面。

### 5.6 工作台卡片系统

- 工作台采用统一卡片系统，而不是为不同角色做不同页面模板。
- 卡片应分为三类：
  - 固定卡片：所有用户都看到，例如快速发起、最近项目、最近会话
  - 权限卡片：只有具备相应权限或角色才出现，例如审批堆积、组织风险热区
  - 可配置卡片：用户可自行隐藏、排序或折叠，但不能突破权限边界
- 卡片内容必须支持明确的数据作用域：
  - `我的`
  - `我负责的`
  - `部门`
  - `全组织`
- 如果某张卡片没有权限或没有数据，应优先隐藏或降级，不要用大量空壳占位。

### 5.7 项目空间固定骨架

- 所有正式 `Project` 页面必须共享同一项目空间骨架。
- 不允许每个项目子页都重新发明导航。
- 固定骨架至少包含：
  - 项目顶栏：项目名、状态、当前阶段、主动作
  - 项目状态条：阻塞、待审批、风险、活跃 Team
  - 二级导航：总览、协作台、看板、时间线与风险、产物、设置
  - 内容区：当前子页主体
- 二级导航顺序固定，不因角色变化而重排。
- 权限影响的是“能否进入”和“显示内容深度”，不是导航结构本身。

## 6. 视觉方向建议

虽然当前产出的是线框，但后续视觉设计应坚持以下方向：

- 气质更接近“企业控制塔”，不是聊天 SaaS
- 信息密度高于普通 AI 对话产品
- 用块级卡片、状态色和分区来强化治理感
- 重要状态必须可扫描：阻塞、审批、风险、运行中、已归档

## 7. 响应式规则

- 平板：侧边栏可折叠，右栏改为抽屉
- 手机：主内容单列，详情全部转全屏抽屉或二级页面
- 移动端优先保留：列表、状态、主动作
- 移动端可弱化：复杂三栏、长表格、多图表并列
