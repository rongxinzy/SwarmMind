# 页面地图与关键流程

> 日期：2026-03-27
> 用途：明确页面全集之间的跳转关系，避免线框只是一堆孤立页面

## 1. 顶层页面地图

```text
工作台
  |- 新建对话
  |- 最近记录
  |- 待审批入口
  |- 项目快捷入口

探索技能
  |- MCP
  |- Skills
  |- Bindings

项目
  |- 项目列表
  |- 新建项目
  |- 项目空间
      |- 总览
      |- 协作台
      |- 看板
      |- 时间线与风险
      |- 产物
      |- 设置

治理
  |- 审批中心

资产
  |- 资源库
  |- 知识库
  |- 定时任务
  |- 设置
```

## 2. 主流程一：探索聊天到正式项目

```text
工作台
  -> 新建对话
  -> ChatSession 形成问题摘要
  -> 用户点击 提升为 Project
  -> 新建项目页预填目标 / 范围 / 约束
  -> 选择 Team
  -> 进入项目总览
  -> 项目协作台开始正式 intake
```

设计要点：

- 提升动作必须显眼，但不强迫。
- `Project` 创建成功后，原 ChatSession 应变成只读来源记录。

## 3. 主流程二：项目启动到持续推进

```text
项目列表
  -> 新建项目
  -> 选择 AgentTeam
  -> 系统自动挂载默认 WorkflowTemplate
  -> 项目空间 / 总览
  -> 项目协作台
  -> 自动生成任务骨架 / 看板 / 初始 artifact
  -> 项目看板 / 时间线 / 产物库持续更新
```

设计要点：

- 用户不应在创建阶段被迫重复选择“Team”和“几乎同义的工作流模板”。
- `TeamInterfaceAgent` 的 intake 结果要回写到 Overview 与 Kanban。

### 3.1 项目空间固定导航

```text
项目空间
  |- 总览
  |- 协作台
  |- 看板
  |- 时间线与风险
  |- 产物
  |- 设置
```

设计要点：

- 这六项构成固定二级导航。
- 新能力优先挂在现有子页中，不新增平行 tab。
- 审批不是项目空间里的常驻 tab，而是跨项目治理入口；项目内通过状态卡和抽屉回看。

## 4. 主流程三：审批暂停与恢复

```text
项目内操作触发高风险能力
  -> 项目状态变为 等待审批
  -> 审批中心收到审批项
  -> 审批人打开审批详情
  -> 批准 / 拒绝 / 要求补充
  -> AuditLog 写入结果
  -> 项目恢复推进或进入阻塞解释态
```

设计要点：

- 审批对象必须是具体请求，不是模糊聊天消息。
- 拒绝后要保留替代建议和影响说明。

## 5. 主流程四：能力治理到项目生效

```text
技能中心
  -> 配置 MCP
  -> 安装 / 绑定 Skill
  -> 绑定到 Team / Project
  -> 项目设置页可见已启用能力
  -> 项目协作台实际调用
  -> 审批中心 / 运行日志展示调用结果
```

设计要点：

- “配置能力” 与 “在项目里生效” 必须形成闭环。

## 6. 主流程五：权限化工作台巡视与下钻

```text
工作台
  -> 按权限看项目组合状态
  -> 查看阻塞项目 / 待审批 / 风险热区
  -> 点入单个项目
  -> 查看项目总览 / 看板 / 审批 / artifact
  -> 回到工作台继续横向比较
```

设计要点：

- 工作台承担驾驶舱角色，但不替代项目页本身。
- 高权限用户可快速下钻，但不应被迫进入执行细节。
