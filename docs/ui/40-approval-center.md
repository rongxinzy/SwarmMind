# 审批中心

> 日期：2026-03-27
> 用途：定义跨项目审批收件箱和审批详情页

## 1. 页面一：审批中心

### 1.1 页面卡片

- 页面名：`审批中心`
- 目标用户：审批人、项目负责人、管理员
- 页面目标：统一处理待审批请求，支持批准、拒绝、补充信息
- 页面入口：工作台待审批卡、项目阻塞卡
- 核心动作：
  - 浏览待审批项
  - 打开审批详情
  - 批准 / 拒绝 / 要求补充

### 1.2 信息结构

- 顶部：筛选器
- 左侧：审批列表
- 右侧：审批详情

### 1.3 桌面端线框

```text
+------------------------------------------------------------------------------------------------+
| 审批中心                                              [待处理] [全部] [按项目] [按风险]       |
+--------------------------------+----------------------------------------------------------------+
| 审批列表                       | 审批详情                                                       |
| CRM / Internal KB 访问         | Project：新一代 CRM 平台建设                                   |
| 招聘 / Offer 发放审批          | Team：软件开发 Team                                            |
| GitLab 写权限审批              | 触发阶段：权限与知识库接入                                     |
|                                | 请求内容：访问 internal-kb 行业资料                            |
|                                | 目的：生成客户行业知识辅助                                     |
|                                | 影响范围：报表、客户画像、行业分析                             |
|                                | 风险等级：high                                                 |
|                                | [批准] [拒绝] [要求补充]                                       |
+--------------------------------+----------------------------------------------------------------+
```

### 1.4 关键交互

1. 点击审批项查看请求上下文、风险和影响范围。
2. 选择 `要求补充` 时，填写补充问题并回流项目协作台。
3. 批准或拒绝后，列表即时更新，项目状态同步变更。

### 1.5 状态设计

- 空状态：展示“当前没有待处理审批”
- 加载状态：列表和详情骨架
- 成功状态：审批操作后给出审计确认
- 错误状态：审批提交失败允许重试
- 权限不足状态：只读用户可查看审批记录但不能操作

### 1.6 移动端规则

- 详情页改为全屏页
- 主操作固定底部

## 2. 页面二：审批详情抽屉

### 2.1 页面卡片

- 页面名：`审批详情`
- 目标用户：审批人
- 页面目标：在不离开当前上下文的情况下完成审批
- 页面入口：审批列表、项目阻塞提示、artifact 引用
- 核心动作：
  - 理解请求
  - 查看来源 run 和 artifact
  - 作出审批决定

### 2.2 桌面端线框

```text
+--------------------------------------------------------------+
| 审批详情                                                  X |
|--------------------------------------------------------------|
| 请求：Internal KB 访问                                      |
| 来源项目：新一代 CRM 平台建设                                |
| 来源任务：行业知识增强                                       |
|                                                              |
| 请求原因                                                     |
| 为补充客户行业资料，需要读取企业内部知识库的行业分析文档      |
|                                                              |
| 相关 artifact                                                |
| - 风险清单 v0.2                                              |
| - 行业字段映射草案                                           |
|                                                              |
| 影响评估                                                     |
| 批准后将恢复被阻塞的报表和客户画像工作流                      |
|                                                              |
| 决策说明 [..............................................]    |
|                                         [拒绝] [批准]        |
+--------------------------------------------------------------+
```

### 2.3 关键交互

1. 从项目中打开时，抽屉覆盖当前页，不破坏原上下文。
2. 审批说明强制要求填写，便于进入 `AuditLog`。

### 2.4 状态设计

- 空状态：无
- 加载状态：抽屉内部骨架
- 成功状态：显示“已写入审计日志”
- 错误状态：提交失败不关闭抽屉
- 权限不足状态：主按钮禁用并显示原因
