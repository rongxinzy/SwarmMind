# SwarmMind Agent Instructions

## Frontend Component Rules

### Component Priority

When building or modifying the React frontend under `ui/src/`:

1. **AI Elements first**: Always prefer components from `@/components/ai-elements/`. This project is built around [ai-elements](https://elements.ai-sdk.dev/) and the AI chat interface must use its primitives (`Message`, `Conversation`, `PromptInput`, `Reasoning`, `Tool`, `Artifact`, `Confirmation`, `Suggestion`, `Task`, etc.).
2. **shadcn/ui second**: If ai-elements does not provide a suitable primitive, use the shadcn/ui components under `@/components/ui/` (Button, Card, Input, Dialog, etc.).
3. **No custom ad-hoc styling**: Do not implement the same UI with raw `<div>` + Tailwind classes when an ai-elements or shadcn/ui component exists for that purpose. Custom markup is only allowed for truly unique layouts that have no matching component.

### Styling Constraints

- Do not introduce arbitrary Tailwind utility combinations to replicate what an existing component already does.
- Custom CSS classes or inline styles are forbidden unless there is no equivalent component primitive and the design cannot be expressed through component props.
- Layout wrappers (flex containers, page shells) are allowed, but they should be minimal and not duplicate component responsibilities.

### Skill Activation

Before editing frontend components, ensure the relevant skills are active:

- `ai-elements` — for all AI chat / message / tool / artifact / confirmation / prompt-input related work.
- `shadcn/ui` — for general UI primitives when ai-elements does not cover the use case.

If a skill is not active, ask the user to activate it before proceeding, or load it via the project skill registry at `.agents/skills/`.

### Scope

These rules apply to all files under `ui/src/components/` and `ui/src/hooks/` that produce UI. Backend Python code and pure utility modules are exempt.

## Design System

### Color Tokens

All colors flow from a single root palette defined in `ui/src/index.css`. The design is cold-toned, near-white, low-contrast — zero saturation interference. Every visual layer uses luminance differences rather than hue shifts.

| Token | Value | Role |
|---|---|---|
| `--background` | `0 0% 100%` (#ffffff) | Main chat area background |
| `--sidebar` | `0 0% 97%` (#f7f7f7) | Sidebar background |
| `--sidebar-accent` | `240 6% 93%` (#ededee) | Selected sidebar button background |
| `--border` | `240 7% 93%` (#ededef) | Global separator / input border |
| `--muted` / `--secondary` | `240 6% 93%` | Secondary surfaces |
| `--foreground` | `0 0% 10%` (#1a1a1a) | Primary text (near-black) |
| `--muted-foreground` | `240 3% 44%` (#8a8a8e) | Secondary text |
| `--primary` | `0 0% 10%` | Button fill (near-black, not a color) |
| `--ring` / `--focus-glow-inner` | `217 100% 85%` (#b2d1ff) | Focus ring inner |
| `--focus-glow-outer` | `210 67% 96%` (#ecf4fc) | Focus ring outer glow |
| `--input-shadow` | `220 8% 91%` (#e7e7e7) | Input box shadow |
| `--radius` | `8px` | Global border radius |

**Dark mode** equivalents are defined under `.dark` in the same file — all tokens invert to dark grays while preserving the same luminance-difference layering.

**NEVER use raw Tailwind color values** (e.g., `bg-blue-500`, `text-gray-600`). Always use semantic tokens (`bg-primary`, `text-muted-foreground`, `border-border`). The only exception is the focus-glow component styles in `index.css`.

### Typography

**Font stack**: Zero-request cross-platform system fonts. No Google Fonts dependencies.

| Token | Stack |
|---|---|
| `font-sans` (UI default) | `ui-sans-serif, system-ui, -apple-system, BlinkMacSystemFont, "Segoe UI", Roboto, "Noto Sans", Ubuntu, Cantarell, "Helvetica Neue", sans-serif, "PingFang SC", "Microsoft YaHei", "Source Han Sans SC", "Noto Sans SC"` |
| `font-serif` (long-form text) | `"Averia Serif Libre", "LXGW WenKai", serif` |
| `font-mono` (code) | `"Maple Mono", "Maple Mono CN", ui-monospace, "SFMono-Regular", "SF Mono", Menlo, Consolas, "Liberation Mono", monospace` |

**Rendering baseline** (applied to `html` in `index.css`):
- `font-feature-settings: normal`
- `font-variation-settings: normal`
- `text-size-adjust: 100%`
- `text-autospace: normal`
- `tab-size: 4`
- `-webkit-tap-highlight-color: transparent`
- `-webkit-font-smoothing: antialiased` (on textareas)

### Input (PromptInput)

Follow the **ai-elements reference structure** exactly:

```
PromptInput
├── PromptInputBody
│   └── PromptInputTextarea  (bare, no custom styling)
└── PromptInputFooter
    ├── PromptInputTools       (action buttons)
    └── PromptInputSubmit
```

**Container** (on `PromptInput` className):
- `rounded-[20px]` large radius
- `border border-[rgba(0,0,0,0.12)]` very light border
- `bg-white` fixed white background
- `flex flex-col` layout, `min-h-[140px] max-h-[300px]`
- `.focus-glow` class for focus state

**Focus glow** (`.focus-glow` in `index.css`):
- `transition: none` — **zero animation, instant response**
- On `:focus-within`: `border-color: rgba(0,87,255,0.35)` + three-layer blue box-shadow
- Inner `InputGroup` focus ring is suppressed at the CSS level

**Textarea**: The bare `PromptInputTextarea` uses ai-elements defaults. Do NOT add custom className or style to it. The only exception: if the InputGroup's internal focus ring reappears, suppress it via the form-level CSS rule `.focus-glow [data-slot="input-group"] { border-color: transparent !important; box-shadow: none !important; }`.

### Layout Constraints

**Sidebar**: Uses shadcn sidebar with `collapsible="icon"` mode. The sidebar gap div uses `--sidebar-width: 16rem`. Because this project runs Tailwind v3.4, the TW4 syntax `w-(--sidebar-width)` does NOT compile. The fix is applied at the consumer level: `<main>` gets `md:ml-[var(--sidebar-width)]` to push content past the fixed sidebar.

**Main content**: Must include `min-w-0` to prevent flex child overflow. All chat content and the input box share a `max-w-3xl mx-auto` centering constraint so their vertical centerlines align.

**Centering reference frame**: Always center relative to the **visual main area** (viewport minus sidebar), NOT the full viewport. Verify with browser DevTools if content appears offset.

### Tailwind Compatibility

This project uses **Tailwind v3.4** (`tailwind.config.js`). The shadcn sidebar component source uses Tailwind v4 syntax:
- `w-(--sidebar-width)` → does NOT work in TW3 (use `w-[var(--sidebar-width)]` instead)
- `group-data-[…]` variants → NOT supported in TW3 (must use custom CSS or workaround)

When adding new shadcn components, always verify the generated CSS includes the expected rules. If a utility class compiles silently to nothing, it is likely a TW4 syntax that needs a TW3 equivalent.

### Known Issues

- **Sidebar gap div**: The shadcn sidebar's gap placeholder uses TW4 syntax throughout. The workaround (`ml-[var(--sidebar-width)]` on `<main>`) is applied, but if the sidebar component is updated from upstream, this fix may need to be re-verified.
- **InputGroup double ring**: The ai-elements `PromptInput` internally renders an `InputGroup` which carries `border` and `focus-visible:ring-*` classes from shadcn. These produce a visible second ring on textarea focus. Suppressed via the `.focus-glow [data-slot="input-group"]` CSS rule. Do not remove that rule.
